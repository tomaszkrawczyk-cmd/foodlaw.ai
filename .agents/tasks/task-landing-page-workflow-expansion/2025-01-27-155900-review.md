# Bilingual landing page, EU scraper expansion, and workflow overhaul (v2)

The five issues from the v1 review have been addressed in commit 842150d. The waitlist form now hides on submit and shows a success message (though emails are still discarded client-side). `search_acts()` routes through the shared HTTP client. The validation step lost `continue-on-error` and the PR creation step is gated on `steps.validate.outcome == 'success'`. The user-agent rotation default flipped to `False`. The validator now checks `.json` files and warns on empty directories.

**Watch for:** Waitlist form still discards submitted emails — the fix is purely cosmetic (confirmed); `--fail-on-empty` flag exists in validate_content.py but is not passed in the workflow, so empty directories still pass silently in CI (confirmed); validation gate can block PR creation when all scrapers fail and produce no output, since validator exits 0 on empty input without `--fail-on-empty` — meaning the gate is a no-op in the all-scrapers-failed scenario (confirmed).

## High-level view

The waitlist fix addressed the visible symptom (unconditionally shown success text) but not the underlying problem: submitted emails go nowhere. The form now `preventDefault()`s, hides itself, and reveals the success message — functionally identical to `/dev/null` for the business. The text "Powiadomimy Cie" (we'll notify you) is an explicit promise that cannot be fulfilled without a backend.

The workflow validation gate is now enforcing: a validation failure blocks PR creation. However, the validator is never invoked with `--fail-on-empty`, so the scenario where every scraper fails (all have `continue-on-error: true`) produces empty directories, the validator finds zero files, and exits 0. The step summary reports "validation: success" which is misleading. The flag was built specifically for this scenario but never wired in.

The `search_acts()` fix and user-agent default flip are clean.

## Waitlist form still discards emails

The fix introduced a new concern: the form now actively misleads users into thinking their email was collected.

```javascript
form.addEventListener('submit', (e) => {
  e.preventDefault();
  form.classList.add('hidden');
  successMsg.classList.remove('hidden');
});
```

The `preventDefault()` suppresses the native submit, the email value is never read from the input, and no fetch/XHR call is made. This is worse than the v1 state in one respect: before, the form at least tried to POST (albeit to `#`); now it explicitly swallows the submission while displaying a success message.

## `--fail-on-empty` gap between script and workflow

The workflow invocation:

```yaml
- name: Walidacja pobranych plikow
  id: validate
  run: |
    python scripts/validate_content.py \
      --input ./output/ \
      --input ./orzecznictwo/ \
      --verbose
```

No `--fail-on-empty` flag is passed. If all seven scrapers fail and produce zero output files, the validator exits 0 and the gate evaluates to "success". The gate blocks bad content (files that fail UTF-8 or size checks) but cannot detect missing content without the flag. The distinction matters for observability: a workflow run where every scraper fails should not report validation as successful.
