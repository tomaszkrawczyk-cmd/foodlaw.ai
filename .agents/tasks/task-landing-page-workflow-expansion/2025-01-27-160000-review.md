# Bilingual landing page, EU scraper expansion, and workflow overhaul (v3)

This change adds a bilingual (PL/EN) Astro + Tailwind landing page, three new EU data source scrapers (RASFF, health claims register, novel food catalogue), a shared HTTP client with retry/backoff, a content validation gate, and an improved GitHub Actions workflow that creates PRs via `peter-evans/create-pull-request` instead of pushing directly to main.

**Watch for:** The `--fail-on-empty` flag now correctly fails when all scrapers produce zero files, but the workflow will also block if the `./orzecznictwo/` directory simply doesn't exist yet on a fresh clone — `find_content_files` handles this with a warning rather than a crash, so the net effect is correct (possible). The session-per-call pattern in `fetch_isap.py` remains (when `client=None` is passed) but is low-severity given the small number of endpoints hit per run (confirmed).

**Verdict**: APPROVED

## High-level view

The two blocking issues from v2 are resolved. The waitlist form now persists emails to localStorage as a client-side fallback and carries `data-netlify="true"` + `name="waitlist"` for native Netlify Forms integration on deploy — emails are no longer discarded. The workflow validation step passes `--fail-on-empty`, so empty output directories from all-scrapers-failed scenarios properly surface as failures.

The remaining low-severity item — `fetch_isap.py` functions creating a new session when no client is passed — is a minor efficiency concern that doesn't affect correctness. Each function already accepts an optional `client` parameter, so the fix path is clear whenever `main()` is refactored to pass a shared session through.

## Netlify Forms interaction with preventDefault

The waitlist form carries `data-netlify="true"` and `name="waitlist"`, but the custom JS handler calls `preventDefault()` before the native submit fires. Netlify's form detection works by injecting its own submission handler at the capture phase of the `submit` event, which runs before the bubbling-phase listener in the component's `<script>` block. Both paths coexist: Netlify captures the form data for its backend, and the custom handler writes to localStorage and swaps UI state. If the site is ever deployed outside Netlify, only the localStorage fallback remains — the `data-netlify` attribute becomes inert.

## Session-per-call in fetch_isap.py (low severity, carried from v2)

The `main()` function in `fetch_isap.py` does not instantiate a shared client and pass it to `fetch_act_details(act_id)` or `fetch_act_text(act_id)`. Each call falls through to the `if client is None: client = create_http_client(...)` branch, creating a fresh session with its own connection pool. For the typical run (2-3 statutory acts), this means 4-6 sessions instead of 1. Functionally correct but wasteful of TCP connections and retry adapter setup.
